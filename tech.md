# AndroidClaw 技术分享：从行业思路到手机端 Vibe Coding 实践

## 1) 项目发起思路

AndroidClaw 的起点并不是“做一个聊天机器人”，而是想把 **LLM 变成真正的手机执行代理**。也就是说，模型不仅理解需求，还要在真实设备上执行操作、观测结果、继续决策。

这个思路来自近两年业内多个方向的收敛：

- 开源 Coding Agent 在“规划-执行-反思”闭环上的工程实践
- 端侧 Agent 在“工具化能力抽象”上的可扩展做法
- 手机助手类产品在“可观测、可恢复、可持续运行”上的经验

我们并没有复刻某一个项目，而是把这些思路落成了一个 Android 原生工程：能运行、能调试、能扩展、能持续迭代。

## 2) 当前项目支持的功能特性

### 2.1 Agent 与执行框架

- 多轮 Agent loop（规划、调用工具、观测、继续执行）
- 记忆机制与压缩机制（含 LLM 辅助压缩）
- 运行元信息记录（耗时、迭代、Token 用量）
- 执行过程可视化（过程消息与悬浮窗）

### 2.2 手机操作能力（核心）

- 无障碍驱动的 UI 操作：点击、长按、滑动、滚动、输入、IME 回车
- 全局动作：返回、Home、最近任务
- APP 启动：按包名打开
- 观测能力：
  - `phone_get_ui_snapshot_compact`（结构化快照，低 token）
  - 截图 data URL / 截图文件导出（调试）
  - 屏幕文本兜底
- 新增节点化操作：`phone_click_node`、`phone_input_node`
- 新增截图坐标映射点击：`phone_click_coordinates_from_screenshot`

### 2.3 渠道与集成

- 统一 Channel 抽象层：入站消息、出站消息、会话持久化、Agent 回答复用同一条链路
- 本地对话页作为一个 channel 实现
- 飞书 channel：长连接接入、去重、健康状态、通道日志

### 2.4 产品化与工程化细节

- 首次权限引导与必需权限门控
- 统一设置页（模型 / 飞书 / Agent）
- 图片持久化、可缩放查看
- 运行日志查看与删除

## 3) 关键技术与设计选择

### 3.1 为什么改成“结构化快照优先”

一开始直接把截图发给模型，虽然通用，但成本高：

- token 消耗大
- 速度慢
- 对复杂页面有时不稳定

所以当前策略改为：

1. 先取无障碍结构化快照（低 token）
2. 能在快照上定位就走节点化操作
3. 快照不够时再用截图
4. 再不行才用文本兜底

这在工程上更可控，也更接近“可长期运行的手机执行代理”。

### 3.2 为什么做 Channel 抽象

如果每个渠道（本地、飞书、未来更多渠道）都各写一套“落库 + 调 agent + 回发”的逻辑，维护成本会快速失控。

统一抽象后，渠道侧只负责：

- 如何接收消息
- 如何发送回复

核心执行链路复用同一套代码，稳定性和可测试性明显更好。

### 3.3 为什么保留可视化执行过程

Agent 的问题不是“是否能回答”，而是“是否真的在执行”。

可视化进度（思考、工具、状态）+ 结构化日志可以让我们快速定位：

- 是模型决策问题
- 是工具能力问题
- 还是渠道/权限/设备问题

## 4) 关于 Vibe Coding：这次实践的体会

这个项目的代码与文档，基本都在手机端场景下通过 **vibe coding** 持续迭代完成。实践下来最明显的感受是：

- 对中高复杂度工程改造（跨层联动、重构、问题排查）已经具备可用性
- 只要需求边界清晰、验证闭环及时，迭代速度明显高于传统方式
- 文档、代码、调试日志可以同步推进，而不是开发完再补文档

当然，vibe coding 不是“自动正确”。它更像一个高带宽协作模式：

- 人负责目标、约束与验收标准
- AI 负责实现速度、重构执行、细节补全

在这个项目里，我们对这种能力是明确认可的：它已经足够支撑一个真实可运行、可演进的 Android Agent 工程。
