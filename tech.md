最近我在做一个项目，叫 AndroidClaw。

如果用一句话说它是干嘛的：

## 一、项目背景：我最想验证的是 Agent 自举开发到底行不行

这个项目最核心的动机，其实不是“再做一个聊天助手”，而是想验证一件事：

> 能不能用 AI Agent 来开发 AI Agent，并且持续扩展 Agent 能力？

这里说的“自举”，不是单指执行闭环，而是“Agent 参与构建 Agent 本身”。

过去很多 Agent 能力是在电脑环境验证的，工具链也相对完整。
但放到手机端，问题一下子就变了：

- 手机界面是动态的
- 操作环境更复杂
- 观测信息噪声更大
- 网络和渠道状态更不稳定

所以我想做的不是一个“看起来很聪明的聊天 UI”，而是一个在手机端可持续迭代的 Agent 原型：一边用它做开发，一边把它能力继续往上长。

---

## 二、实现方法：用手机端 Vibe Coding，一步步把链路跑通

整个过程其实很朴素：

1. 在手机上装 Termux
2. 用 `proot-distro` 跑 Ubuntu
3. 在 Ubuntu 里安装 OpenCode
4. 后续环境配置、代码开发、调试、编译、Git 操作尽量都在这条链路里完成

示例命令大概是这样：

```bash
# Termux
pkg update && pkg upgrade -y
termux-change-repo
pkg install -y proot-distro

# Ubuntu
proot-distro install ubuntu
proot-distro login ubuntu
apt update && apt install -y curl sudo
curl -fsSL https://linuxmirrors.cn/main.sh | bash
```

这个方法最大的好处就是反馈很快：

- 想到一个点就能马上改
- 改完马上编译验证
- 不用等回到电脑再开始干活

从工程角度看，这条链路也比较接近标准 Linux 开发流程，后续迁移到 PC 或 CI 都比较自然。

## 三、目前支持的功能特性

现在这个项目已经不只是 demo 了，核心链路已经能稳定跑起来。

### 1) Agent 执行闭环

- 支持多轮执行（规划、调用工具、根据结果继续决策）
- 支持暂停和恢复
- 有执行过程状态展示（包括悬浮窗/过程消息）

### 2) 手机操作能力（基于无障碍）

- 点击、长按、滑动、输入、全局动作
- 打开 App、读取当前界面
- 截图调试（支持 data URL 和文件落盘）

### 3) UI 理解能力

- 以结构化 UI 快照为主
- 支持节点级操作（比如 `click_node`、`input_node`）
- 截图作为兜底，不是默认主路径

### 4) 渠道能力

- 本地对话和飞书通道统一在 channel 层处理
- 飞书支持长连接接入
- 有去重和基本健康状态可观测

### 5) 产品化配套

- 统一设置页（模型、飞书、Agent）
- 图片消息持久化和大图查看
- 运行日志查看与清理

## 四、几个我觉得很有用的小巧思

这些点看起来不大，但对体验影响非常明显。

### 1) 结构化快照优先，而不是每轮都看截图

如果每轮都让模型看大图，token 成本和延迟都很高。
改成结构化 UI 快照优先后，模型看到的是可操作信息，执行效率和稳定性都会更好。

### 2) 截图坐标映射点击

截图链路里我做了图片压缩算法，先把大图压到更适合推理的尺寸和质量，再送给模型。
这个改动带来两个直接收益：更省资源、推理更快。

另外，截图压缩后图上坐标和真实屏幕坐标会有偏差，我补了坐标映射，保证兜底点击依然准确。

## 五、未来可迭代方向

我接下来最想做的是：把“能用”继续推进到“更好用、更便宜、更稳定”。

### 方向 1：端侧轻量模型决策

- 用云端大模型产高质量执行轨迹
- 蒸馏轻量模型做端侧下一步动作选择
- 形成“端侧优先，云端兜底”的混合模式

目标是更低延迟、更低成本、更好的弱网可用性。

### 方向 2：更强的结构化 UI 理解

- 提升节点检索稳定性
- 做更可靠的滚动搜索策略
- 增强复杂控件/自绘区域的混合观测能力

### 方向 3：更完善的通道可靠性

- 接入更多消息渠道
- 把 ACK、重试、超时、死信做成更清晰的状态机
- 继续补强去重和可观测能力

## 六、Vibe Coding 体验：真的快，但前提是你把目标说清楚

我的真实感受可以概括成一句话：

**Vibe Coding 很能打，但不是“自动成功”。**

它特别适合：

- 跨文件改造
- 功能链路打通
- 文档和代码同步推进
- 快速迭代 + 及时验证

但它也很吃前提：

- 需求边界要清楚
- 验收标准要明确
- 每次改动都要有回归验证

只要这三点到位，产出质量是能控的，迭代速度也会非常可观。

最后回到最开始的问题：

我这次想验证的“用 Agent 开发 Agent、并持续扩展 Agent 能力”这件事，至少在手机端这条工程路径上，已经证明是可以一步步做实的。
