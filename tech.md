# 我用手机 Vibe Coding 复刻了一个“手机版 OpenClaw”

## 背景与目标

这次分享面向团队内部，核心问题不是“能不能做一个聊天机器人”，而是：

> 能否在手机端构建一个可运行、可观测、可持续迭代的 LLM 执行代理？

AndroidClaw 的目标是把 LLM 从“回答问题”推进到“执行手机任务”：

- 理解用户意图
- 规划执行步骤
- 调用手机能力完成操作
- 根据观测结果持续修正

项目思路借鉴了业内一些开源与产品化方向（如 OpenClaw / NanoBot / 豆包手机助手 / AutoGLM），但落地时重点是工程可用性与可维护性，而非复刻某个单点实现。

## 一、手机端 Vibe Coding 的实际操作方式

在谈技术细节前，先给出这次实践中“手机端研发环境”是如何搭起来的，方便团队复现。

### 1.1 环境链路

整体链路是：

- 手机上安装 Termux 作为入口
- 在 Termux 里用 `proot-distro` 运行 Ubuntu
- 在 Ubuntu 里安装 OpenCode
- 后续环境配置、代码开发、编译、Git 操作，尽可能由 OpenCode 完成

### 1.2 关键步骤（示例）

```bash
# Termux
pkg update && pkg upgrade -y
termux-change-repo
pkg install -y proot-distro

# 安装并进入 Ubuntu
proot-distro install ubuntu
proot-distro login ubuntu

# Ubuntu 内换源
apt update && apt install -y curl sudo
curl -fsSL https://linuxmirrors.cn/main.sh | bash

# 安装 OpenCode（以官方文档为准）
# 之后的开发/编译/git 由 OpenCode 持续执行
```

### 1.3 这种方式的价值

- 需求-实现反馈链路短：想到即改、改完即验
- 环境贴近标准 Linux 工程流程，迁移成本低
- AI 可承担大量重复工程动作，人更聚焦目标和验收标准

## 二、系统架构与功能范围

当前项目可按五层理解：

- **Agent Core**：规划-执行循环、记忆压缩、工具调度、技能/子代理
- **Tool Layer**：手机/文件/命令/网络工具 + MCP 外部工具桥接
- **UI 理解层**：结构化快照优先，截图/文本兜底
- **Channel 层**：本地频道 + 飞书频道的统一收发链路
- **存储与可靠性**：Room 持久化、守护服务、日志与健康检查

功能上目前覆盖：

- Android 无障碍驱动的点击、滑动、输入、全局动作
- 结构化 UI 快照（低 token）与节点级操作
- 图片预览、消息过程追踪、执行状态悬浮窗
- 飞书长连接消息接入与通道化处理
- 运行日志查看与排障

## 三、关键技术决策（内部重点）

### 3.1 观测策略从“截图优先”改为“结构化快照优先”

早期方案主要靠截图发给模型，问题是：

- token 开销高
- 速度慢
- 观测噪声大

当前策略：

1. `phone_get_ui_snapshot_compact`（低 token）
2. `phone_click_node` / `phone_input_node` 做节点级交互
3. 快照不够再用截图
4. 截图仍不够再用文本兜底

收益是明显的：token 成本下降、动作稳定性提升、可解释性更强。

### 3.2 坐标点击的压缩映射问题

图片压缩后，模型在“压缩图”上的坐标不能直接用于真实屏幕。

因此新增了 `phone_click_coordinates_from_screenshot`：

- 输入：截图坐标 + 截图宽高
- 逻辑：比例映射到真实屏幕
- 输出：真实点击

这个改造解决了“看着点对了但实际点偏”的一类隐蔽问题。

### 3.3 Channel 抽象带来的收益

如果每个渠道都单独实现“入站->落库->调 Agent->回发”，很快会分叉。

现在通过 `ChannelEngine` 统一这条链路：

- 本地对话页可视作 Local channel
- 飞书是 Feishu channel（长连接）

这样做后：

- 业务链路复用
- 故障点集中
- 新渠道接入成本下降

### 3.4 可观测性优先于“看起来能跑”

这次实践里，可观测性不是附加项，而是主设计目标之一：

- 执行悬浮窗：思考/工具/当前状态
- 对话内过程消息聚合（单任务一条）
- token/耗时/迭代元信息可回看
- 运行日志（含飞书事件、去重、回发结果）

这让我们能快速区分问题属于模型决策、工具能力还是渠道链路。

## 四、飞书接入现状与重构点

飞书这块已经从“功能可用”重构到“可维护”：

- 事件长连接接入
- 去重（message_id 窗口）
- 健康状态可视化（连接状态、最近事件、最近错误）
- 通道日志留痕（INCOMING/OUTGOING/DEDUP/ERROR）

内部建议：下一步把飞书消息 ACK 与重试策略再做显式状态机，进一步提升极端网络下的稳定性。

## 五、研发方法总结：Vibe Coding 在团队内的定位

这次项目代码与文档都通过手机端 vibe coding 持续迭代完成。团队角度看，它的价值不在“替代工程师”，而在于提升迭代吞吐。

### 5.1 适合的任务类型

- 多文件联动改造
- 明确验收标准的功能实现
- 可快速验证的排障与重构
- 文档与代码同步演进

### 5.2 风险与边界

- 需求不清时，AI 会放大歧义
- 缺少观测与测试时，容易“看起来完成”但不可维护
- 需要明确代码审查与验收门禁

### 5.3 团队实践建议

- 把“可验证结果”写进需求
- 保持小步提交与持续编译验证
- 让日志、追踪、健康检查成为默认能力

## 结论

这个项目的关键收获是：

1. 手机端 Agent 可以做到“可运行 + 可观测 + 可迭代”
2. 结构化 UI 理解是降低成本、提升稳定性的关键抓手
3. Vibe coding 在工程团队中已经具备实用价值，前提是边界清晰、验证闭环到位

换句话说，这不是一次“炫技实验”，而是一次可复用的工程化实践。
